# 2026-06-15 캡슐화

## 오늘 배운 내용

- 메소드를 경유한 필드 조작
- getter : 읽기 전용을 구현할때 사용
- setter : 쓰기 전용을 구현할 때 사용(잘 안 쓴다고함)
- getter/setter의 메리트
- Read Only,Write Only 필드의 구현
- 필드의 이름 등, 클래스의 내부 설계를 자유홉게 변경 가능
- 필드로의 액세스를 검사 가능

## 기억할 것

- 클래스의 액세스 제어의 지정방법과 범위
- package private 아무것도 쓰지 않고 자신과 같은 패키지에 소속된 클래스 접근 가능 범위
- public public기술방법을 사용하며 모든 클래스에 접근 가능한 범위
- 비 public 클래스의 특징
- 클래스의 이름은 소스 파일명과 달라도 된다
- 1개의 소스 파일에 여러개 선언 해도 된다
- 필드는 전부 private
- 메소드는 전부 public 가 정석
- 별다른 이유가 없으면 public

## 정리

### 캡슐화의 개요

- 캡슐화를 하여 멤버나 클래스로의 접근을 제어할 수 있음
- 특히, 필드에 “현실세계에서 있을 수 없는 값"이 들어가지 않도록 제어

### 멤버에 대한 접근 지정

- private 지정된 멤버는, 동일 클래스내에서만 접근 가능
- package private (아무것도 지정 안된) 지정된 멤버는, 동일 패키지내의 클래스에서만 접근 가능.
- public 지정된 멤버는, 모든 클래스에서 접근 가능

### 클래스에 대한 접근 지정

- package private (연산자 없음) 으로 선언 된 클래스는, 동일 패키지내의 클래스에서만 접근 가능.
- public 지정된 클래스는, 모든 클래스에서 접근 가능

### 캡슐화의 정석

- 클래스는 public, 메소드는 public, 필드는 private 로 지정
- 필드에 접근하기 위한 메소드로서 getter 나 setter를 준비
- setter 내부에서는 인수의 타당성 검사를 수행

## 실습 코드

```java
package com.survivalcoding;

public class Wand {
    private String name;
    private double power;

    public String getName() {
        return name;
    }

    public void setName(String name) {
        if (name == null) {
            throw new IllegalArgumentException("지팡이의 이름은 null일 수 없습니다.");
        }
        if (name.length() < 3) {
            throw new IllegalArgumentException("지팡이의 이름은 반드시 3문자 이상이어야 한다.");
        }
        this.name = name;
    }

    public double getPower() {
        return power;
    }

    public void setPower(double power) {
        if (power < 0.5 || power > 100.0) {
            throw new IllegalArgumentException("지팡이의 마력은 0.5이상 100.0이하여야 합니다.");
        }
        this.power = power;
    }
}


```

## 어려웠던 점

- “throw new IllegalArgumentException(“메시지");”의 문법을 새로 알게 되었다.

## 해결 방법

- 새로알게된 문법을 좀더 찾아봤다

## 내일 더 공부할 것
