# 2026-06-22 추상클래스와 인터페이스

## 오늘 배운 내용
- 추상 클래스를 안쓸 때의 문제점
  - 자식이 오버라이드를 안하면 메서드는 호출되는 데,
    부모의 빈 attack() 이 실행돼서 아무 일도 안 일어남.
  - 주석으로 알려줘도 컴퓨터가 강제로 막아주지는 않음
  - 철자를 틀려도 문제
    - attack이 아니라 attck이라면 사람은 어떤 의미인 지 알지만,
      컴퓨터 입장에서는 다른 메서드라서 오버라이드가 되지 않음
      - @Override를 붙이면 이런 실수를 줄일 수 있음
  - Character를 직접 만들어 버릴 수 있음
    - Character 는 전사, 마법사도 아닌 캐릭터의 공통 설계도
    - Character c = new Character();
      - 캐릭터라는 설계도만 보고 캐릭터를 만들 수 없음.
      - 전사, 마법사 같은 구체적은 캐릭터가 있어야 함.
      - 그래서 자식 클래스를 만들 때만 사용해야 함
- 추상 클래스
- 즉, 미완성 부모 설계도
  - public abstract class Character // 추상 클래스
  - public abstract void attack(Kinoko kinoko); // 추상 메서드
    -  일반 메서드는 public void run(){
           System.out.println("도망쳤다");
       } 
       중괄호 안에 내용이 있다
  -  추상 메서드는 public abstract void attack(Kinoko kinoko);
  - 세미 콜론으로 끝남
  - 공격 기능은 반드시 있어야 한다.
  - 하지만 공격 내용은 자식 클래스가 정해라.
  - 추상 메서드가 있으면 클래스도 추상 클래스여야 함
    - public abstract class Character { // abstract 를 써야함 
          public abstract void attack(Kinoko kinoko);
      }
  - 추상 클래스는 오버라이드를 강제함
    - Character를 상속받으면 attack()을 반드시 만들어야 함
      - public class Dancer extends Character {
           public void dance() {
             System.out.println("춤을 춘다");
          }
        }
        - attack()을 안 만들었기 때문에 오류. 
      - public abstract class Dancer extends Character {
            public void dance() {
              System.out.println("춤을 춘다");
            }
        }
      - 추상 클래스로 만들면 가능
  - 추상 클래스는 미완성 설계조라서 직접 만들 수 없음
    - Character c = new Character(); // 불가능
    - Hero hero = new Hero(); // 가능
    - 즉, 추상 클래스는 부모 역할만 한다.
      직접 객체로 만들 수 없다.
  - 다계층 추상 상속 구조
    - Monster 
      ㄴ WalkingMonster   ㄴ FlyingMonster
          ㄴ Goblin ㄴ Wolf     ㄴ DeathBat
    - Monster는 모든 몬스터의 공통 부모
    - WalkingMonster는 걸어다니는 몬스터
    - FlyingMonster는 날아다니는 몬스터

- 인터페이스
  - 인터페이스는 추상 메서드만 가진 추상 클래스처럼 생각하면 됨
    - 추상 클래스로 쓸 때,
      - public abstract class Creature {
           public abstract void run();
        }
    - 인터페이스로 쓸 때,
      - public interface Creature {
            public abstract void run();
        }
  - 인터페이스에는 일반 필드는 안 두는 게 기본
  - 대신 상수는 둘 수 있음
    - public interface Creature {
        // public static final이 생략 
        double PI = 3.14;

        void run();
      }
  - 인터페이스 활용 예시
    - public interface Attackable {
         void attack(Slime slime);
      }
      - 공격 가능함
    - public interface Healable {
         void heal(Hero hero);
      }
      - 치유 가능함
  - 클래스가 인터페이스를 사용하려면 implements를 사용
    - public class Hero implements Attackable {
        protected String name;
        protected int hp;

        @Override
        public void attack(Slime slime) {
          System.out.println(name + "이 공격했다");
          slime.hp -= 10;
        }
      }
      - Hero는 Attackable 약속을 지킨다.
      - 그래서 attack()을 반드시 가지고 있어야 한다.
  - 인터페이스의 효과
    - 공통 메서드를 강제로 만들게 할 수 있음
      - Attackable을 구현하면 무조건 attack()을 만들어야 함.
    - 그 클래스가 어떤 기능을 가지고 있는지 보장할 수 있음
      - 어떤 클래스가 Healable을 구현했다면, 
        이 클래스는 heal() 기능이 있다고 믿을 수 있음
  - 인터페이스는 여러 개 구현 가능
    - 자바에서는 부모 클래스를 여러개 가질 수 없지만,
      인터페이스는 여러개 구현할 수 있음
      - public class Hero implements Attackable, Moveable {
        }
        - Hero는 공격도 가능하고, 이동도 가능하다.
      - 즉, 클래스 상속은 "부모 하나"만 가능하지만,
        인터페이스는 "능력 여러 개"를 붙일 수 있음
  - 인터페이스끼리도 상속 가능
    - public interface Creature {
        void run();
      }
    - public interface Human extends Creature {
        void talk();
        void watch();
        void hear();
      }
      - Human은 Creature의 run()도 가지고,
        자기 자신의 talk(), watch(), hear()도 가진다는 뜻
      - 클래스는 run(), talk(), watch(), hear()를 전부 만들어야 함
  - extends와 implements
                                  키워드      여러 개 가능?
    - 클래스가 클래스를 상속         extends      불가능
    - 클래스가 인터페이스를 구현     implements    가능
    - 인터페이스가 인터페이스를 상속   extends      가능
  - Can do 원칙
    - 상속은 보통 is-a 관계
      - Hero is a Character
        - 용사는 캐릭터다.
    - 인터페이스는 Can-do 관계
      - Hero can attack
        - 용사는 공격할 수 있다.
  - 상속과 구현을 동시에 사용 가능
    - extends 먼저, implements 나중
      - public class Wizard extends Hero implements Attackable, Healable {
        }
## 기억할 것

- 추상 클래스는 미완성 부모 설계도
  - abstract class Character
    - extends -> ~이다
      - Wizard extends Hero
      - 마법사는 용사의 일종이다.
- 추상 메서드는 내용이 없는 메서드
  - {} 가 없고 ;로 끝남
- @Override를 붙이면 철자 실수를 컴파일 단계에서 확인 가능
- 인터페이스는 기능 약속
  - interface Attackable
    - implements -> ~할 수 있다.
      - Wizard implements Healable
      - 마법사는 치료할 수 있다.
  - 여러 개 구현 가능
    - Hero implements Attackable, Moveable
- 추상 클래스와 인터페이스 차이
  - 추상 클래스: 공통 필드나 일반 메서드도 가질 수 있는 미완성 부모 클래스
  - 인터페이스: 주로 기능을 강제하기 위한 약속
  - 추상 클래스는 extends로 상속
  - 인터페이스는 implements로 구현
- 추상 클래스: 부모 설계도인데 아직 미완성
- 추상 메서드: 이 기능은 꼭 있어야 하는 데 내용은 자식이 정해야 함
- 인터페이스: 이 기능을 할 수 있다고 약속

## 실습 코드

```java
package com.survivalcoding;

public abstract class TangibleAsset {
  private final String name;
  private final int price;
  private final String color;

  public TangibleAsset(String name, int price, String color){
    this.name = name;
    this.price = price;
    this.color = color;
  }

  public String getName(){
    return name;
  }

  public int getPrice(){
    return price;
  }

  public String getColor(){
    return color;
  }

}
public class Book extends TangibleAsset {
  private final String isbn;

  public Book(String name, int price, String color, String isbn) {
    super(name, price, color);
    this.isbn = isbn;
  }

  public String getIsbn() {
    return isbn;
  }
}
public class Computer extends TangibleAsset{
  private final String makerName;

  public Computer(String name, int price, String color, String makerName){
    super(name, price, color);
    this.makerName = makerName;
  }

  public String getMakerName(){
    return makerName;
  }
}

```
